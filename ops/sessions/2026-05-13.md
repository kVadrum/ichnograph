## Picked (and why)
HTML entity decoding has the broadest hit rate of the cleanup candidates:
any README authored as HTML-tolerant markdown (or rendered through a
tool that escapes ampersands) carries entities in the very first
paragraph that summary extraction reads. The change is parallel in
shape to the prior ~20 summary-cleanup sessions: a small helper plus
duplicated entries in both `notes.ts` and `readme.ts` (matching the
workspace convention of "duplicate code freely; canonical-home is for
rules, not code"). Decoding runs *after* the HTML-tag strip so an
encoded-literal `&lt;b&gt;` survives the stripper and collapses
correctly in the final pass — order matters and is documented inline.

## Diff summary
The `decodeEntities` helper matches a single named/decimal/hex entity
per replacement callback, so `&amp;lt;` decodes only once and emerges as
`&lt;` rather than `<` — matching HTML5 spec behavior. The named-entity
table is a curated allowlist (XML core + common typographic chars +
smart quotes); unknown names stay literal so prose tokens like
`&foo;` aren't guessed at. Surrogate-range and out-of-range numeric
entities are rejected. Reviewer should sanity-check the regex bounds
(name length 2–16, decimal 1–7 digits, hex 1–6) — chosen to fit all
real entities while bounding catastrophic-backtracking risk.
