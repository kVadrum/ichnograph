## Picked (and why)
Went with the blockquote-marker fix. The notes and readme detectors have
been accruing per-construct strippers over the past few weeks (HTML
comments, footnotes, autolinks, list markers, closing-hash sequences),
and blockquote was a conspicuous gap — a `STATE.md` whose first line is
`> Status: active` surfaced the marker verbatim. CommonMark §5.1 covers
the optional space (`>foo`) and nesting (`>>`) cases, both of which the
regex now handles. The fix is per-line so multi-line blockquotes in a
README still join cleanly into one sentence rather than each surfacing
their own `>`.

## Diff summary
The fix adds blockquote-marker stripping to the two detectors that
extract one-line summaries from notes (STATE/TODO/etc.) and from
READMEs. Notes get a single regex inserted into `firstMeaningfulLine`
as a sibling to the list-marker pass — capture group becomes the body
forwarded to `stripInlineMd`. Readme path strips per-line inside the
paragraph collector, so multi-line `>` blocks still join into a single
prose sentence rather than surfacing each marker. A bare `>` line is
treated as a paragraph terminator (mirrors the existing empty-line
break). Reviewer should sanity-check the readme behavior on a multi-line
blockquote — that's the one place the per-line dequote could surprise
if a line ends mid-sentence.
