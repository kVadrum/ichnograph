## Picked (and why)
The image-syntax bug is symmetric across the two near-identical strip
helpers (`stripMd` in `readme.ts`, `stripInlineMd` in `notes.ts`),
trivially testable, and visible whenever a README or `STATE.md`
header includes a badge / logo image — exactly the surfaces ichnograph
exposes first. Fits a single cycle cleanly. Fix: insert an image
regex (`!\[([^\]]*)\]\([^)]+\)` → `$1`) before the existing link
regex on both helpers, with a comment noting the order dependency.
Empty-alt images (`![](url)`) collapse to the empty string, which is
what we want for badges with no caption.

## Diff summary
The new image regex uses `[^\]]*` (zero or more) where the link regex
uses `[^\]]+` (one or more), so empty-alt images are matched and
removed entirely — without the looser quantifier they would fall
through to the link regex as `[](url)`, which doesn't match the link
regex either, and the literal `![](url)` would survive. Order
matters: image must run before link, otherwise the link regex eats
`[alt](url)` first and the image regex never sees a match. Both new
helpers carry an inline comment explaining the order dependency.
Subtleties for a reviewer: nested brackets in alt text (e.g.
`![[1]](url)`) still aren't handled — same limitation the link regex
has always had — and badges with empty alt followed by other text get
a leading space that `.trim()` removes for the readme path but is
preserved mid-line by `.split(' ').join(' ')`-style joining in the
paragraph build (the existing test covers the resulting "Real
description." case).
