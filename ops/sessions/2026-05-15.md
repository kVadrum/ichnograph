## Considered
- Backslash-escape handling in inline-markdown stripping (`\*literal\*`) —
  thorny: the existing italic regex mis-matches `\*foo\*` to `foo`, and
  a clean fix needs sentinel substitution or per-delimiter negative
  lookbehinds. Too much regex surface for one cycle.
- Deno `deno.json` name/version surfacing — parallel to the recent PHP
  composer.json version work; small, contained, clear parity gap.
- Elixir `mix.exs` app+version extraction — viable but requires parsing
  the Elixir keyword-list inside `def project do`; more brittle than
  reading a JSON field.
- Dart `pubspec.yaml` version — line-regex against `version:` would
  work; smaller value than deno.json (Deno is a more-used target).
- Badge-cluster suppression in README first-paragraph extraction —
  heuristic-heavy, higher regression risk than tonight's budget allows.

## Picked (and why)
Deno's `deno.json` already gates the language detection but reports
`name: null, version: null` unconditionally, even though JSR-published
packages routinely declare both at the top level (`{"name":
"@scope/pkg", "version": "1.2.3"}`). That's the same parity gap that
got closed for composer.json last week. The change uses the existing
`readJsonSafe` + `stringOrNull` helpers — no new parsing surface — and
fixes an adjacent latent bug while I was in there: the `manifest`
field was hard-coded to `'deno.json'` even when only `deno.jsonc`
existed, so the report mis-named the source file.

## Diff summary
The Deno branch in `detectStack` now does an actual JSON parse instead
of hard-coding nulls. When both `deno.json` and `deno.jsonc` exist,
`deno.json` wins — matching Deno's own resolution order. JSONC parsing
isn't implemented: a `deno.jsonc` with C-style comments returns null
from `readJsonSafe` and the surface degrades to null name/version,
which is no worse than the previous behavior. Reviewer check: the
`manifest` string now reflects the file actually inspected, which is a
small JSON-schema observable change — additive (existing consumers
that expected `'deno.json'` literally see the same string in the
common case; only the deno.jsonc-only case differs, and that case
previously reported a file that wasn't being read).

## Followups / blockers
- JSONC support (deno.jsonc with comments) — would need a tolerant
  parser or a comment-stripper preprocessor. Not urgent; Deno itself
  treats the two as interchangeable but the JSON form dominates in
  practice.
- Deno `imports` map could feed framework detection (e.g. `jsr:@hono/hono`
  → Hono), parallel to the JS framework path. Larger scope; defer.
