## Picked (and why)
The readme paragraph-extraction loop walks consecutive non-blank lines
after the title until it hits a heading, blank, or `===` underline. A
README that leads with a usage snippet — e.g. `# Title` then a fenced
` ```sh\n$ npx mytool\n``` ` block, then the description — would have
its "summary" rendered as the literal fence contents. Same shape of
bug as the one fixed in notes.ts on 2026-04-26, but in the
higher-visibility surface. Fix mirrors the notes one: at the start of
paragraph collection, treat a fenced opener as a "skip past close,
look for prose" event; once the paragraph has begun, treat a fence as
a terminator (like a heading).

## Diff summary
The new branch in the loop runs only when `paraLines.length === 0`,
so the existing terminator semantics are preserved for any fence that
appears mid-paragraph. Inside the skip, we advance `i` past the
matching close fence (matched by `trim().startsWith(marker)`, so the
close can be at any indentation), then advance through any
immediately-following blank lines, then `continue`. The for-loop's
`i++` then lands on the first non-blank prose line. Subtleties for
a reviewer: an unterminated fence (file ends mid-block) walks to EOF
and returns `summary: null`, which is correct; a file that contains
only a fence after the title also returns `null` (covered by the new
third test). All six existing readme tests still pass unchanged.
