## Picked (and why)
Elixir `mix.exs` name + version detection. mix.exs was one of the last
manifests with both fields hard-coded to null, and the recent stack
work has been steadily closing that gap (Gleam, Dart, Deno, PHP).
mix.exs is Elixir code rather than a declarative format, but the
project metadata lives in a stable shape (`def project do [...] end`
with `app:` and `version:` keys), and the `@version` module-attribute
pattern is the canonical idiom for published Hex packages so handling
it pays for most real-world Elixir libraries.

## Diff summary
The `def project do ... end` regex uses non-greedy `[\s\S]*?` up to a
`\n\s*end` boundary, which holds for the typical case of a project
function that returns a literal keyword list with no nested do/end. If
a project body opened a nested `do` block (e.g. an `if` expression),
the scope match could stop short — acceptable because the canonical
shape doesn't do that, and the fallback is null rather than wrong
data. The atom regex for `app:` allows trailing `?` / `!` (valid in
Elixir atoms) but only at the end. The `@version` resolution path
searches the full file for the attribute definition, which is
intentional — module attributes are typically declared above the
`project/0` function, outside the scoped block.
