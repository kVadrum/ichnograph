## Considered
- HTML comment stripping in `stripInlineMd` / `stripMd` — a `STATE.md`
  or `CLAUDE.md` whose first non-blank line is a canonical-home marker
  (`<!-- canonical: ~/dev/ai/prompts/foo.md -->`) currently surfaces
  that comment as the one-line summary instead of digging past it.
- Backslash-escape handling (`\*foo\*` → `*foo*`) — interaction with
  the existing emphasis regexes is non-trivial (the naive italic
  regex still matches inside escaped pairs and yields garbled
  output), so doing this safely needs a sentinel-style two-pass
  approach. Not in scope for a 15-min slot.
- HTML inline tag stripping (`<b>`, `<br/>`, etc.) — risks regressing
  the existing test that says bare placeholders like `<name>` must be
  left alone. Would need a tag whitelist, larger surface.
- Extracting the duplicated `stripInlineMd` / `stripMd` helpers into a
  single shared module — the bodies have been identical and edited in
  lockstep across ~7 sessions, which fits the canonical-home pattern
  in `~/dev/CLAUDE.md`. Skipped this cycle because the autonomous
  prompt flags broad/structural refactors as out-of-scope; better to
  surface as a followup for a human-driven session.
- Pyproject inline-table form (`scripts = { foo = "mod:fn" }`) for
  entrypoints — narrow real-world frequency, deferred.

## Picked (and why)
HTML-comment stripping. It's small (~one regex per helper), covers a
genuinely real case in this very workspace (canonical-home markers
inside `STATE.md` / `CLAUDE.md`-style notes), follows the established
pattern of the recent inline-strip series, and has zero ambiguity:
`<!--...-->` is unique enough that there's no risk of collateral damage
to legitimate prose. The replacement consumes surrounding whitespace
so mid-line comments don't leave double spaces behind. Multi-line
comments are handled in the readme detector via the paragraph-line
joiner; a multi-line comment as the very first lines of a `STATE.md`
is left as a follow-up (would need line-spanning logic in
`firstMeaningfulLine`).

## Diff summary
The fix adds `\s*<!--[\s\S]*?-->\s*` → ` ` as the first replacement in
both helpers. Putting it first means the comment body never participates
in any later regex (so a comment containing `**bold**` or a stray `*`
can't accidentally feed the emphasis pass). The whitespace clamps on
both sides keep mid-line use clean: `Status <!-- TODO --> active` ends
up as `Status active`, not `Status  active`. When a notes file's first
non-blank line is *only* a single-line comment, the strip yields an
empty string and `firstMeaningfulLine`'s existing `length === 0`
continue advances to the next line — so canonical-home markers no
longer hijack the summary.

## Followups / blockers
- `stripInlineMd` (notes) and `stripMd` (readme) bodies are now
  identical 30+ line blocks edited in lockstep across many sessions.
  Worth extracting to a shared helper in a human-reviewed session;
  fits the canonical-home / shared-protocol pattern in `~/dev/
  CLAUDE.md`.
- Multi-line HTML comments at the very top of a notes file (before
  any heading) are still surfaced as the summary for the first line
  of the comment. Fixing this needs line-spanning awareness in
  `firstMeaningfulLine` similar to the fenced-code-block handling.
