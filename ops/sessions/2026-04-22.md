## Picked (and why)
Fixed the README detector's handling of setext-style headings. With input
like `My Tool\n=======\n\nParagraph.`, the old code picked up the title
correctly but the paragraph scan saw the `=====` underline, matched the
separator guard, and returned `summary: null`. Setext headings are the
second-most-common Markdown heading style (and standard in reStructuredText
and many hand-written READMEs), so the miss was a real quality issue for
that corner of the input space. The fix is one conditional line plus two
tests — small, testable, and in keeping with the graceful-degradation
principle.

## Changes
- src/detect/readme.ts — after picking a non-ATX title, skip a following `===`/`---` underline so it doesn't terminate the paragraph scan.
- tests/readme.test.ts — added two tests covering setext H1 (`===`) and H2 (`---`).

## Diff summary
The title-detection loop already had a fallback that treated the first
non-blank non-`#` line as the title. The fix adds a lookahead: if the line
immediately after the chosen title is `===` or `---` (3+ chars), advance
past it before the blank-line skip. That way the paragraph scanner lands
on the real first paragraph instead of bailing out on the underline.
Reviewers should verify that the existing tests (ATX H1, frontmatter,
lowercase filename, inline-markdown stripping) still pass unchanged —
they do — and that the new cases match `Title` + `Summary paragraph.`
as expected.
