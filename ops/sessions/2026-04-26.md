## Picked (and why)
`firstMeaningfulLine` returns the first non-empty line of a notes
file. If that line is the opening of a fenced code block (` ``` `
or `~~~`, optionally with a language tag), the rendered summary
ends up as the fence itself — useless and a touch embarrassing for
files that lead with an example block. The fix is local: detect
fences at the candidate-line stage, skip past the matching close
fence, and continue. Mirrors the project's "graceful degradation"
principle (don't surface garbage; just omit or skip past it).

## Diff summary
The fence regex matches three or more backticks or tildes at the
start of a stripped line (the trim happens upstream in the existing
loop). On a hit, we capture the marker, advance `i`, and walk
forward until a line that — after trimming — `startsWith(marker)`,
which catches the close fence at any indentation. Then `continue`
to the outer loop, which keeps scanning subsequent lines for real
content, applying the existing heading regex and `stripInlineMd`
treatment. No schema impact, no new exports, no new deps. The one
subtlety to check: an unterminated fence (file ends mid-block)
makes the loop run to EOF and return `null`, which is correct —
there is no meaningful line to surface in that case.
